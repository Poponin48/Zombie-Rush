Краткая шпаргалка: что уже настроено и что сделать вручную.

## Правила (уже созданы)

В `.cursor/rules/` лежат три правила:

| Файл | Когда применяется | Смысл |
|------|-------------------|--------|
| `game-design-workflow.mdc` | Всегда | Язык ответов (RU), фокус на геймдизайн, пошаговые инструкции в движке, без костылей, учёт Git |
| `unity-csharp.mdc` | При открытии `.cs` | Чистый C#, отладка, без лишних абстракций, учёт WebGL |
| `unity-engine-steps.mdc` | При открытии сцен/префабов/ProjectSettings | Формат инструкций «что сделать в Unity», типичные ошибки |

Их не нужно редактировать, если не хотите изменить поведение агента.

## Примеры: жёсткие vs гибкие правила (ответ на вопрос 13)

**Жёсткое правило** — агент всегда обязан так делать:
- «Всегда после выдачи нового C# скрипта добавляй блок «Что сделать в Unity» с нумерованными шагами.»
- «Никогда не оставлять пустой catch или игнор null для SerializeField.»

**Гибкое правило** — рекомендация:
- «По возможности используй [Header] и [Tooltip] для полей в Inspector.»
- «Предпочтительно одну ответственность на класс.»

Сейчас в правилах смешано: часть требований жёсткие (язык, формат инструкций, запрет костылей), часть — рекомендации (стиль кода, комментарии). При желании можно усилить или ослабить формулировки в `.mdc` файлах.

## Рекомендуемые настройки Cursor

Открой **Cursor Settings** (Ctrl+,) и проверь:

1. **Features → Chat**
   - Включён ли **Auto-apply** — по желанию; если хочешь меньше ручных правок, можно включить осторожно и откатывать через Git при ошибках.

2. **Models**
   - Для агента можно оставить модель по умолчанию или выбрать более мощную для сложной архитектуры и отладки.

3. **Rules**
   - Правила из `.cursor/rules/` подхватываются автоматически. В настройках можно убедиться, что **Project rules** включены.

Дополнительно в `settings.json` (глобально) при желании можно добавить, например:
- `"editor.formatOnSave": true` — автоформатирование при сохранении (удобно для C#).
- Размер шрифта, тема — по вкусу.

Файл настроек (Windows): `%APPDATA%\Cursor\User\settings.json`.

## .cursorignore (создать вручную, если нужно)

Чтобы Cursor не индексировал тяжёлые папки Unity, в **корне проекта** создай файл `.cursorignore` с содержимым:

```
/Library/
/Temp/
/Obj/
/Build/
/Builds/
/Logs/
/UserSettings/
*.csproj
*.sln
.vs/
```

Если файл не создаётся из-за прав — можно работать и без него; правила от этого не пострадают.

---

## Git

- В корне проекта создан **.gitignore** под Unity (Library, Temp, .vs, логи и т.д.). Коммитить нужно в основном `Assets/`, `ProjectSettings/`, `Packages/manifest.json`.
- **.cursorignore** не заменяет .gitignore: первый ограничивает, что Cursor отдаёт в контекст AI; второй — что попадает в репозиторий.

Рекомендация: сделать первый коммит (`git init`, добавить файлы, commit), затем при каждой новой фиче или скрипте — отдельный коммит, чтобы при багах можно было откатиться.
